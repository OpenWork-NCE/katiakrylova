## Migration could not be run end-to-end

Per the task description, no real Postgres `DATABASE_URI` exists in this environment. The repo only ships `.env.example` with placeholders like `postgres://user:pass@host:5432/db`. `migrate-content.ts` calls `getPayload({ config })` which instantiates `postgresAdapter` and tries to open a pool — without a real DB the script would fail at the first `payload.updateGlobal` call. Running it end-to-end requires:

1. A reachable Postgres instance with the schema initialised (`pnpm payload migrate` or `pnpm dev` first).
2. `.env.local` with real `DATABASE_URI`, `PAYLOAD_SECRET`, (and `BLOB_READ_WRITE_TOKEN` if uploading media).
3. The admin user to upload `coverImage` and `gallery` for each of the 19 projects (Step 5 in brief).
4. The admin user to publish each draft (`draft: true` was used in the seed; covers must be uploaded before publishing).

This is a known manual step that requires user-supplied env vars and admin UI access. Documented here per task instructions.

## Follow-up: Enable drafts on Projects collection (Important finding fix)

`scripts/migrate-content.ts` uses `draft: true` on `payload.create({ collection: 'projects' })` to bypass the required `coverImage` field. In Payload v3, `draft: true` only skips required-field validation at runtime when the target collection has `versions: { drafts: true }` enabled. Without it, the seed would throw `coverImage is required` on the first project despite the script passing typecheck.

### Change

Added the drafts block to `src/collections/Projects.ts:6-10`:

```ts
versions: {
  drafts: {
    autosave: false,
  },
},
```

`autosave: false` — admin saves manually after uploading `coverImage` and `gallery`, then publishes each draft (matches the documented workflow in the Migration section above).

`PortfolioCategories` was inspected (`src/collections/PortfolioCategories.ts`): only has `name`, `slug`, `order` — no required upload fields — so drafts are not needed there and were not added.
